Development Roadmap for OpenAFS

Pending issues for OpenAFS 1.2 releases

The following are items known to be in progress, and, where available, a point of contact.

"CopyOnWrite" bug

The "namei" fileserver (used on Linux but not by default on Solaris, AIX, IRIX or Tru64/Digital Unix) is known to corrupt data under circumstances which have not become entirely clear yet. What is known is that generally after a volume clone (such as is caused by vos backup) a later write will log a "CopyOnWrite" failure, and the corrupted volume will be taken offline. After OpenAFS 1.2.3 a bug was found which at the time was believed to be related, however it has not yet been proven to fix the problem. The fix is, however, included in OpenAFS 1.2.5. If you believe you are experiencing this problem, and especially if you can reproduce it, please contact Derrick Brashear (shadow@dementia.org).

Windows NT server

Had problem with paths in the installation as of 1.2.2b release. Fix integrated but not yet available in a release.

Linux .__afsXXXX file creation

Believed to be introduced with changes to the Linux client in OpenAFS 1.2.3.

Linux vcache NULL pointer dereference

Fix is present in CVS but not in a release yet.

Linux lost wakeups

As issue is known, a proposed fix is on-hand and will be integrated when it can be assured it causes no other problems.

Windows symlink behavior

Known to be flawed; No known fixes.

To be addressed in 1.3 before 1.4 is forked

The following are items which involve fundamental behavior changes or are not expected to be advanced enough to be included in 1.2 releases.

Fakestat

The cache manager can fake mountpoint directory information so clients which stat all of /afs will not need to time out all the cells. Not yet tested on all platforms, and will not fix MacOS X Finder issues.
